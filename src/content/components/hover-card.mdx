A hover card is a [tooltip](/components/tooltip)'s bigger sibling. A tooltip is a scrap of text on
an icon button; a hover card is a *card* — an avatar, a name, a bio — surfaced from a link in running
text. Rather than wire up hover state, delays and a `pointer-events` bridge by hand, this hands the
whole job to the platform with an interest invoker.

## How it works

- `interestfor` on the link points at the card, a `popover="hint"` element. The browser handles
  showing it on hover **and** keyboard focus, the open/close delays, the top layer, and keeping it
  alive while you move the cursor toward it — no `group`/`peer` wiring, no `pointer-events` guard.
- The card declares `[position-anchor:--hc-janedoe]` back at the link's `[anchor-name:--hc-janedoe]`,
  so `[position-area:block-end_span-inline-end]` places it below and to the right.
- `[position-try-fallbacks:flip-block]` keeps it on screen: near the bottom of the viewport the
  browser flips the card above the link on its own. Try it near a screen edge and watch it move.
- The entry/exit animation is the same discrete-transition pattern used across the library —
  `@starting-style` for the opening frame, `transition-behavior: allow-discrete` so it animates out
  before `display` flips back.

## The fallback story

Interest invokers are Chromium-only for now. In other browsers the card simply never appears — the
link still works and still navigates, so nothing is lost. If you need hover previews for everyone
today, the same idea works with a `group/card` wrapper and `group-hover`/`group-focus-within` (see
the plain-CSS [tooltip](/components/tooltip) for that technique), accepting that a hand-positioned
card can't reroute at a viewport edge.
